# 候选池后二次压力突破监控

## 背景

上游系统发出的 `candidate_score_passed` 只表示项目评分合格。V2 不再把该事件直接当作买入信号，而是在 V2 信号模块中继续跟踪该 CA 的最新 Birdeye K 线，等它真正出现回测中的“突破压力带买入”场景后，再向交易模块发出买入信号。

## 业务规则

- V2 信号模块订阅上游候选池 Redis Pub/Sub channel：`solana_scalper:candidate_pool`。
- 收到 `candidate_score_passed` 后，把 CA 写入 Redis 监控池，状态为 `watching`。
- 监控池每 2 秒轮询一次 active CA，直接调用 Birdeye 原始数据源查询 1m 最新 K 线；该路径不使用项目级 K 线缓存。
- `watching` 状态下，如果最新 K 线收盘市值小于 `15000`，该 CA 从监控池移除，不再继续监控。
- `watching` 状态下，如果候选进入之后出现“突破压力带买入”场景，V2 发出标准 `TradeSignalMessage` 买入信号，`strategyCode=breakout_band_follow`，并把状态改成 `bought`。
- `bought` 状态继续监控卖点，即使最新市值低于 `15000` 也不移除，避免已买入仓位无人退出。
- 卖点复用回测 `breakout_band_follow` 规则：下一根跌回压力带上沿、硬止损、止盈、动态止损；实时监控不使用样本结束 timeout 卖出。
- 买入和卖出信号都发布到 V2 标准交易信号 Redis channel，由交易模块按当前 `paper/live` 模式执行。

## Redis 状态

- Active set：`solana_meme_v2:signal_monitor:active`
- 单 CA hash：`solana_meme_v2:signal_monitor:candidate:{ca}`
- 幂等 key：`solana_meme_v2:signal_monitor:emitted:{signalId}`

单 CA hash 记录 CA、symbol、上游 payload、候选进入时间、状态、买入 signalId、买点、压力带信息等。服务重启后会从 active set 恢复未结束监控。

## 配置示例

```yaml
redis:
  enabled: true
  channel: "solana:meme:signals:pressure_breakout"
  consumer_channel: ""

signal:
  candidate_monitor_enabled: true
  candidate_channel: "solana_scalper:candidate_pool"
  poll_interval_seconds: 2
  interval: "1m"
  min_market_cap: 15000
  lookback_bars: 120
  redis_key_prefix: "solana_meme_v2:signal_monitor"

trade:
  enabled: true
  signal_consumer: true
```

## 与交易模块的关系

交易模块只消费标准 `TradeSignalMessage`。候选池评分合格事件由信号模块处理，不再由交易模块直接转换成买入。
